# 企業文書AIエージェント化における「実行層の欠落」とその解法

> 対象読者：AIを活用した業務変革（AI-BPR）を日本企業に提案するコンサルタント・PoC担当者

---

## 1. 問題の本質：なぜ「文書を扱うAI PoC」は最後の一歩で止まるのか

日本企業のAI-BPR案件において、最も頻出する業務要件は次の2種類に集約される：

**① 文書改訂型**
規程・仕様書・マニュアルをAIが読み込み、改訂が必要な箇所を特定し、修正を加えた上で正式フォーマットのまま納品する。

**② 検査・記録型**
ある基準書や点検項目に照らし合わせて文書・データを検査し、結果を所定の様式（Excel台帳・チェックシート等）に記録・追記する。

どちらも「AIが判断した結果を、元のファイルに書き戻す」という工程が不可欠だ。

しかし現状のAIビルダー（Dify、LangFlow、Copilot Studio等）はここで止まる。**LLMの出力はMarkdownやJSONであり、元のWordやExcelファイルに戻る経路を持たない。**

結果として「PoC段階では参考情報としてMarkdownで出力します」という説明に終始し、業務フローへの実質的な組み込みが先送りになる。

## 2. 既存アプローチの限界マップ

| アプローチ                                                | できること                                                              | できないこと                                                                                                                                                                            |
| --------------------------------------------------------- | ----------------------------------------------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Dify / LangFlow / n8n**                           | ワークフロー編成、LLM呼び出し、HTTP連携                                 | Word/ExcelのフォーマットをそのままAIが編集して返却することはできない（Markdown出力どまり）                                                                                              |
| **Microsoft Copilot Studio**                        | Power Automateと連携してExcelセルの更新やWordテンプレート差し込みは可能 | M365/SharePoint 環境前提。任意のローカルWordファイルへの外部的編集は不可。オンプレ文書への対応はゲートウェイ設定が必要で複雑                                                            |
| **OpenAI Code Interpreter / Claude Code Execution** | PythonでWord/Excelを操作するコードを生成・実行できる                    | **文書データが外部サーバー（OpenAI/Anthropic）を通過する。** インフラ・金融・公共系の日本企業がPoC適用を躊躇する最大の理由になる。加えて、LLMが書くコードは非決定的で監査できない |
| **RPA（UiPath等）**                                 | 画面操作ベースで既存業務をそのまま自動化                                | AI理解を持たない機械操作。UI変更で壊れる。開発・維持コストが高い                                                                                                                        |
| **python-docx / openpyxl の直接実装**               | 完全なファイル操作が可能                                                | ドキュメントを一から再構築するため、スタイル継承・結合セル・ヘッダーフッター・採番系等が崩れる。本番品質を出すには大規模実装が必要                                                      |

## 3. このツールが埋めるギャップ

```
[既存の壁]
Dify等のOrchestrator → LLM → Markdown出力 → ⛔ ここで止まる

[このツールがいる場合]
Dify等のOrchestrator
  └─ POST /ingest      → DOCX/XLSX をLLMが読める構造化JSONに変換（フォーマット情報を保持）
  └─ LLM               → テキスト理解・判断・編集箇所の特定（LLMはテキストだけを触る）
  └─ POST /generate/from-edit → 元のDOCXに外科的に差分を書き戻す（書式・スタイルは一切変えない）
                         → ダウンロード可能なWord文書として返却 ✅
```

このツールの本質的な役割は**「LLMの意思決定と、実際のファイル操作の間に立つ実行層（Execution Layer）」**だ。

LLMは「何を変えるか」だけを判断し（old_text → new_text のJSONを出力）、ファイルを触ることはない。ファイル操作は決定的（deterministic）なコードが行うため、監査・再現・ロールバックが可能になる。

---

## 4. データ主権の問題：なぜ「外部コード実行」では代替できないか

OpenAIのCode InterpreterやAnthropicのコード実行サンドボックスは技術的には文書操作が可能だ。しかしエンタープライズ文書の観点からは致命的な問題がある：

**文書の内容が外部サーバーを経由する。**

東北電力のような電力インフラ企業、あるいは銀行・官公庁・防衛関連企業においては、業務文書の内容を外部クラウドに送ることが契約上・規制上・内規上の制約になるケースが非常に多い。

このツールを使う場合：

- **LLMに渡るのはテキスト内容のみ**（段落テキスト・表のセル値）
- **バイナリ（DOCX/XLSXファイル本体）はローカルに留まる**
- **ファイルの読み書きはすべてオンプレ環境で完結する**

LLMにはファイルのバイナリを渡していない。テキストだけを渡してテキストだけを返させ、バイナリへの適用はローカルで行う。これが「外部AIと自分たちの文書を安全に組み合わせる」ための正しい境界線の引き方だ。

## 5. Agentic・MCP時代における位置づけ

現在このツールは2つのインターフェースを持つ：

- **HTTP API**：Dify・LangFlow等のオーケストレーターから呼び出す
- **MCP Server**：Claude Desktop・その他MCPクライアントから直接ツールとして呼び出す

今後、AgentのSkillやToolkit化が進む中で、このアーキテクチャは以下のように展開できる：

```
┌──────────────────────────────────────┐
│   オーケストレーター層（今後多様化）    │
│  Dify / LangGraph / AutoGen          │
│  Claude Agent SDK / Copilot Studio   │
└──────────────┬───────────────────────┘
               │ ツール呼び出し（HTTP / MCP / Skill）
               ▼
┌──────────────────────────────────────┐
│  このツール（実行層）                  │
│  - ingest: 文書 → 構造化JSON          │
│  - generate/from-edit: 差分 → 文書   │
│  ローカル動作・データ外部流出なし       │
└──────────────────────────────────────┘
```

**重要な点：** このツールはオーケストレーターに依存しない。Difyに組み込まれたワークフローとして動くこともできるし、Claude DesktopのMCPツールとして動くこともできるし、将来のAgentフレームワークからSkillとして呼ばれることもできる。

ツールの安定性とオーケストレーターの選択は切り離されている。これがこのアーキテクチャの最も重要な設計判断だ。

## 6. PoC提案における訴求ポイント

日本企業へのAI-BPR提案でこのツールを位置づける場合、以下の言い方が効く：

> 「このシステムでは、AIは文書の"意味"だけを判断し、文書の"形式"には一切触れません。修正の適用はすべて決定的なプログラムが行います。したがって、AIが何かを誤解しても書式が崩れることはなく、変更内容はすべてJSONとして記録・再現・ロールバックが可能です。」

これは日本企業が求める「説明可能性」「監査可能性」「担当者による承認フロー」との親和性が高い。

## 7. 現時点での局限性（正直に伝えるべきこと）

| 制約                             | 内容                                                                                                                                                          |
| -------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **段落単位の編集のみ**     | 現状の `generate/from-edit`は段落全体の置換。run（文字）レベルの部分的な太字・色変更には未対応                                                              |
| **old_textの完全一致依存** | 段落テキストとの完全一致で対象を特定するため、同一テキストが複数存在すると最初のものが変更される。stable ID（ingest時に付与する段落ID）による定位に移行すべき |
| **複雑なWord構造**         | セクション区切り・ヘッダーフッター内のテキスト・テキストボックス内容は現状非対応                                                                              |
| **Excelは値更新のみ**      | 数式・結合セル・条件付き書式・印刷設定への対応は限定的                                                                                                        |
| **スケール未検証**         | 大量ファイルの並列処理・負荷試験は未実施                                                                                                                      |

---

## 8. 結論

AIビルダーやオーケストレーターは「LLMへの指示と結果の受け渡しを整理する」ための道具だ。しかしそれだけでは、日本企業の「元のファイルを変えて返してほしい」というシンプルかつ本質的な要件を満たせない。

この実行層が存在することで初めて：

- PoC出力が「参考Markdown」から「実際に使える正式文書」になる
- データが外部サーバーを経由しないため、インフラ・金融・公共系の顧客でも適用できる
- LLMの交換（Gemini→Claude→GPT-4o）がオーケストレーター側の変更だけで済む

**「LLMが判断し、このツールが実行し、人が承認する」** という3層構造が、日本企業のAI-BPR実装における現実解だ。
